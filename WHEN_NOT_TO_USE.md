# 언제 Ensembra를 쓰지 말아야 하는가

> **토큰 절약의 가장 큰 효과는 "Ensembra를 쓰지 않는 결정"에서 나옵니다.**

Ensembra는 6명의 에이전트가 토론하는 **심의 지향 도구**입니다. 오타 수정에 6명의 토론은 과잉이며, 단순 질문에 5-phase 파이프라인은 낭비입니다. 이 문서는 언제 Ensembra를 **쓰지 말아야 하는지** 판단 기준을 제시합니다.

---

## 1. 즉시 수정이 나은 경우 (Ensembra 생략)

다음 작업은 Ensembra 없이 Claude Code로 **직접 수정**하는 것이 효율적입니다.

### 범위 기준

- **2~3줄 이하 변경**: 오타, null 체크, 상수값 교정
- **단일 파일 내 작은 수정**: 함수 내부 로직 1~2군데
- **주석·문서 수정**: README, 인라인 주석, JSDoc
- **명백한 상수 교정**: 환경변수 값, URL, 타임아웃 숫자
- **UI 텍스트 변경**: 버튼 라벨, 에러 메시지 문구

### 예시

| 요청 | 판단 |
|------|------|
| "로그 메시지에 오타 고쳐줘" | 직접 수정 |
| "TIMEOUT 값 30초 → 60초" | 직접 수정 |
| "주석 오역 수정" | 직접 수정 |
| "버튼 텍스트 '확인' → '저장'" | 직접 수정 |

### 단, 다음 신호가 있으면 즉시 수정 금지

- `/auth/`, `/payment/`, `/migrations/`, `/middleware/` 경로
- `.env`, `config/`, schema 파일
- "인증", "결제", "세션", "권한" 등의 도메인
- 공통 모듈(`commons/`, `shared/`, `lib/`) 수정
- 아무리 작은 변경이라도 위 조건에 해당하면 **ops-safe 프리셋 이상 사용**

## 2. Claude 직접 대화가 나은 경우 (Ensembra 생략)

### 읽기·이해 작업

- **코드 설명 요청**: "이 함수 뭐 하는 거야?"
- **원인 분석**: "왜 500 뜨지?", "로그 해석해줘"
- **개념 질문**: "이 라이브러리 어떻게 동작해?"
- **옵션 탐색**: "이 문제 해결 방법 몇 가지 보여줘"

### 단순 실행

- **명령어 실행 확인**: "npm install 해줘"
- **로그 tail / grep**: "에러 로그 찾아줘"
- **파일 검색**: "이 함수 어디에서 쓰여?"

### 예시

| 요청 | 판단 |
|------|------|
| "이 에러 왜 나는 거지?" | Claude 직접 대화 |
| "DB 인덱스 걸린 컬럼 보여줘" | Claude 직접 대화 |
| "CSRF 토큰이 뭔지 설명해" | Claude 직접 대화 |

## 3. Ensembra를 써야 할 때

다음 조건 중 하나라도 해당하면 Ensembra 사용이 정당화됩니다.

- **회귀 위험이 있는 수정** — 기존 테스트가 깨질 가능성
- **여러 모듈 걸친 변경** — 횡단 영향
- **새 기능 추가** — 요구사항 해석이 필요
- **인증·결제·세션·스키마 건드림** — 치명 신호 도메인
- **공통 모듈 수정** — `commons/`, `shared/`, `lib/`
- **외부 API 계약 변경** — 브레이킹 체인지 가능성
- **보안 관련 변경** — 권한·암호화·토큰

## 4. 판단 플로우차트

```
요청 받음
   │
   ├─ 읽기·질문·이해만 필요?
   │     → Claude 직접 대화 (Ensembra 생략)
   │
   ├─ 2~3줄 이하 변경 + 위험 키워드/경로 없음?
   │     → 직접 수정 (Ensembra 생략)
   │
   ├─ 위험 키워드/경로 1개 이상 감지?
   │     → /ensembra:run ops-safe (최소 안전 경로)
   │
   ├─ 회귀 위험 있는 버그 수정?
   │     → /ensembra:run bugfix
   │
   ├─ 새 기능·소규모?
   │     → /ensembra:run feature --profile=pro-plan
   │
   └─ 인증·결제·스키마·중대 설계?
         → /ensembra:run feature --profile=max-plan
```

## 5. 위험 키워드·경로 체크리스트

아래 중 **하나라도** 요청 또는 변경 대상에 포함되면 Ensembra 사용 (최소 `ops-safe` 프리셋).

### 🔴 Critical 도메인 (무조건 Ensembra 필수)

- 인증·로그인·로그아웃·세션·토큰
- 결제·빌링·환불·정산
- 권한·역할·ACL·admin
- DB schema·migration·DROP·ALTER
- 환경변수·시크릿·API 키

### 🟠 High 도메인 (변경 2줄 초과 시 Ensembra 권장)

- 외부 API 호출·webhook
- 트랜잭션·락·동시성
- 캐시·큐·메시징
- 배포·CI/CD·롤백

### 🟡 경로 기반 체크

| 경로 패턴 | 기본 행동 |
|---------|---------|
| `/auth/`, `/session/`, `/middleware/` | Ensembra 필수 |
| `/payment/`, `/billing/` | Ensembra 필수 |
| `/migrations/`, schema 파일 | Ensembra 필수 |
| `.env*`, `/config/` | Ensembra 권장 |
| `/api/`, `/controller/` | 2줄 초과 시 권장 |
| `/utils/`, `/helpers/` (공통 모듈) | Ensembra 권장 |
| `/docs/`, 주석·README | 생략 가능 |

## 6. 실전 예시 모음

### Case A — 타이포 수정 (생략)

**요청**: "login 페이지 제목 '로그인 화면' → '로그인'"

**판정**: UI 텍스트, 2줄 이하, 위험 경로 아님 (표시 문구만)

**처리**: 직접 수정. `find`·`Edit` 2번으로 끝.

### Case B — 겉보기 단순, 실제 위험 (Ensembra 필수)

**요청**: "null 체크 하나만 추가해줘 (auth/validateToken.js)"

**판정**: 경로 `/auth/` 치명 신호. 단 2줄이라도 Ensembra 필수.

**처리**: `/ensembra:run ops-safe` — qa·security·devils-advocate가 회귀·우회·블래스트 반경 검증.

### Case C — 질문만 (생략)

**요청**: "우리 프로젝트 인증 방식이 JWT야 세션이야?"

**판정**: 읽기만. 변경 없음.

**처리**: Claude 직접 대화. `Grep` 1번으로 답변.

### Case D — 버그 수정, 회귀 위험 높음 (bugfix 프리셋)

**요청**: "주문 목록 페이징 쿼리가 가끔 중복 레코드 반환"

**판정**: DB 쿼리 수정, 회귀 위험 있음, 다중 호출자 가능성.

**처리**: `/ensembra:run bugfix --profile=pro-plan` — qa가 엣지케이스, security가 SQL injection 점검.

### Case E — 신규 기능, 결제 연관 (max-plan)

**요청**: "쿠폰 할인 적용 로직 추가"

**판정**: 결제 도메인, 신규 기능, 치명 신호.

**처리**: `/ensembra:run feature --profile=max-plan` — 전원 토론, 전원 감사, opus 최종 판정.
