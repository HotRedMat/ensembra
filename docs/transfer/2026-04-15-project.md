## 0. 요약 (Summary)

Ensembra는 **1인 개발자를 위한 Claude Code 멀티에이전트 오케스트레이터 플러그인**이다. 6명의 전문 Performer(planner, architect, developer, security, qa, devils-advocate) + 1명의 기록자(scribe)가 5단계 파이프라인(Gather → Deliberate → Execute → Audit → Document)을 통해 구조화된 코드 리뷰와 상호 감시를 수행한다. Reuse-First 교차 정책으로 중복 구현을 방지하고, 합의 임계치(70/40)로 품질을 보증하며, 모든 작업을 자동 문서화한다. 현재 v0.1.0 Gate2 구조 검증이 완료된 상태이며, 실제 런타임 실행 로직 구현이 다음 단계이다.

## 1. 프로젝트 목적 및 목표

**담당 Performer**: planner

### 1.1 핵심 문제

1인 개발자는 코드 리뷰, 설계 검증, 보안 감사를 받을 동료가 없다. 이로 인해:
- 설계 편향이 검증 없이 통과됨
- 보안 취약점이 사각지대에 남음
- 기술 부채가 문서화 없이 축적됨

### 1.2 해결 방향

| 항목 | 내용 |
|---|---|
| **목표** | Claude Code 플러그인으로 멀티에이전트 심의 파이프라인 제공 |
| **핵심 가치** | Reuse-First (재사용 우선), Consensus-Driven (합의 기반), Auto-Documentation (자동 문서화) |
| **대상 사용자** | Claude Code를 사용하는 1인 개발자 |
| **배포 채널** | Claude Code 마켓플레이스 (claude.ai/settings/plugins) |

### 1.3 요구사항 (확정)

- [x] 6명 Performer + 1명 Scribe 에이전트 정의
- [x] 5단계 파이프라인 (Phase 0~4) 설계
- [x] 6개 프리셋 (feature, bugfix, refactor, security-audit, source-analysis, transfer)
- [x] Reuse-First 4개 디바이스 (Inventory, Schema Field, Auto-Disagree, Synthesis Report)
- [x] JSON 스키마 기반 입출력 계약
- [x] 합의 임계치 및 감사 메커니즘
- [ ] 실제 런타임 실행 로직 (Gate3 범위)
- [ ] Ollama/Gemini 트랜스포트 통합 (Gate3 범위)

### 1.4 마일스톤

| Gate | 상태 | 내용 |
|---|---|---|
| Gate1 | ✅ 완료 | 설계 합의, CONTRACT.md, INTERVIEW.md 확정 |
| Gate2 | ✅ 완료 | 플러그인 구조 구축, 에이전트·스킬·프리셋·스키마 전체 작성, 라이브 검증 |
| Gate3 | 🔲 미착수 | 런타임 실행 로직, 트랜스포트 통합, 실제 프로젝트 테스트 |
| Gate4 | 🔲 미착수 | 마켓플레이스 배포, 사용자 피드백 반영 |

### 2.2 파이프라인 아키텍처

```
사용자 요청
    │
    ▼
Phase 0 — Deep Scan (6필수 + 4선택 항목)
    │  구조, 키워드, 콜그래프, 데이터플로, 재사용 인벤토리, 문서 인벤토리
    │  + 테스트맵, git 이력, 의존성, 설정 (선택)
    ▼
Phase 1 — Deliberate
    │  R1: 6 Performer 독립 분석 (병렬)
    │  R2: 상호 검토 + Peer Signature (조건부)
    │  Synthesis: 합의 ≥70% → Plan 확정
    ▼
Phase 2 — Execute
    │  Claude Code(Conductor)가 Plan 실행
    │  외부 Performer는 쓰기 권한 없음
    ▼
Phase 3 — Audit
    │  지정 감사자가 diff ↔ Plan 검증
    │  fail → Rework (최대 2회)
    ▼
Phase 4 — Document
    │  scribe가 Task Report + 추가 문서 생성
    ▼
완료
```

### 2.3 Performer 매트릭스

| 역할 | 기본 모델 | 트랜스포트 | 참여 Phase | 핵심 책임 |
|---|---|---|---|---|
| planner | opus | claude-subagent | 1, 3 | 요구사항 확장, 수용 기준 정의 |
| architect | sonnet | gemini (→claude) | 1, 3 | 모듈 경계, 설계 패턴, 구조 결정 |
| developer | sonnet | claude-subagent | 1, 3 | 구현 계획, 라이브러리 선택, 코드 스타일 |
| security | sonnet | ollama/qwen2.5 (→claude) | 1, 3 | 위협 모델, OWASP, CVE, 시크릿 감시 |
| qa | sonnet | ollama/llama3.1 (→claude) | 1, 3 | 엣지케이스, 회귀 검증, 테스트 커버리지 |
| devils-advocate | haiku | claude-subagent | 1, 3 | 반론, 과잉설계 경고, YAGNI |
| scribe | sonnet | claude-subagent | 4만 | 문서 생성 (토론 불참, Peer Signature 없음) |

### 2.4 합의 메커니즘

- **≥70%**: Plan 확정, Phase 2 진행
- **40~70%**: R3 추가 라운드 또는 사용자 판단
- **<40%**: 중단, 토론 요약 반환
- Peer Signature: R2에서 각 Performer가 다른 모든 Performer에 agree/disagree/abstain 투표

### 2.5 프리셋별 구성

| 프리셋 | Performers | 라운드 | Phase 2 | Phase 3 감사자 | Phase 4 문서 |
|---|---|---|---|---|---|
| feature | 전원(6) | R1+R2+합성 | ✅ | 전원 | Task+Design+Request |
| bugfix | 4명(planner,arch,dev,qa) | R1+합성 | ✅ | qa, security | Task만 |
| refactor | 4명(arch,dev,devils,qa) | R1+R2+합성 | ✅ | arch, devils | Task+Design+Request |
| security-audit | 3명(security,devils,arch) | R1+합성 | ❌ | — | Task만 |
| source-analysis | 3명(arch,security,dev) | R1+합성 | ❌ | — | Task만 |
| transfer | 전원(7, scribe 포함) | R1만 | ❌ | — | 인수인계서(10섹션) |

### 2.6 핵심 설계 결정

| 결정 | 선택 | 근거 (INTERVIEW.md) |
|---|---|---|
| 배포 형태 | Claude Code 플러그인 전용 | Node CLI 병행 불필요, 플러그인이 충분 |
| 구현 언어 | Markdown + YAML (런타임 의존성 없음) | Claude Code 플러그인 스펙이 .md 기반 |

## 3. 빌드, 실행, 개발 환경

**담당 Performer**: developer

### 3.1 런타임 의존성

- **필수**: Claude Code (플러그인 호스트)
- **선택**: Ollama (로컬 LLM — qwen2.5:14b, llama3.1:8b)
- **선택**: Gemini API (무료 티어)
- **빌드 도구 없음**: Markdown/YAML만 사용, 컴파일/번들 불필요

### 3.2 설치

```bash
# 클론
git clone https://github.com/HotRedMat/ensembra ~/.claude/plugins/ensembra

# 플러그인 구조 검증
claude plugin validate ~/.claude/plugins/ensembra
```

### 3.3 선택적 트랜스포트 설정

```bash
# Ollama (security, qa 에이전트용)
ollama serve
ollama pull qwen2.5:14b   # security 에이전트
ollama pull llama3.1:8b    # qa 에이전트

# Gemini (architect 에이전트용)
# 키 발급: https://aistudio.google.com/app/apikey
# /ensembra:config → Transports → Gemini API key 입력
# → ~/.config/ensembra/env (chmod 600)에 저장됨
```

### 3.4 실행

```bash
# 메인 파이프라인
/ensembra:run feature 결제에 쿠폰 할인 추가
/ensembra:run bugfix 체크아웃 경합 조건 수정
/ensembra:run refactor 결제 모듈 분리

# 읽기 전용 분석
/ensembra:run security-audit OWASP Top 10 스캔
/ensembra:run source-analysis 인증 모듈 구조 분석

# 설정
/ensembra:config

# 보고서
/ensembra:report daily
/ensembra:report weekly

# 인수인계서
/ensembra:transfer
/ensembra:transfer src/payment
```

### 3.5 설정 파일 위치

| 파일 | 경로 | 용도 |
|---|---|---|
| 사용자 설정 | `~/.config/ensembra/config.json` | 모델, 라운드, 타임아웃 등 |
| Gemini 키 | `~/.config/ensembra/env` | API 키 (chmod 600) |
| 플러그인 매니페스트 | `.claude-plugin/plugin.json` | 플러그인 구조 선언 |

- 에이전트 정의: Markdown + YAML frontmatter

## 5. 테스트 커버리지

**담당 Performer**: qa

### 5.1 현재 상태

Ensembra는 Markdown/YAML 기반 플러그인으로 **전통적인 유닛 테스트 코드가 없다**. 검증은 Gate2에서 수행한 라이브 통합 테스트로 대체되었다.

### 5.2 Gate2 검증 결과 (CHANGELOG.md)

| 항목 | 결과 | 비고 |
|---|---|---|
| E2E 스모크 테스트 | ✅ Pass | feature 프리셋으로 샌드박스 JS 프로젝트에서 Phase 0~3 실행 |
| 에이전트 개별 검증 (8개) | ✅ Pass | 각 에이전트 라이브 호출, 역할 준수 확인 |
| Reuse-First 정책 적용 | ✅ Pass | Synthesis에서 기존 패턴 재사용 확인됨 |
| 감사 오버라이드 로직 | ✅ Pass | QA rework → 과반 미달 → Pass 처리 |
| 플러그인 매니페스트 검증 | ✅ Pass | `claude plugin validate` 통과 |

### 5.3 신뢰 영역 vs 불안정 영역

| 영역 | 신뢰도 | 이유 |
|---|---|---|
| 에이전트 정의 (agents/) | 🟢 높음 | 라이브 검증 완료 |
| 프리셋 (presets/) | 🟢 높음 | feature 프리셋 E2E 검증 |
| 스키마 (schemas/) | 🟡 중간 | 구조 정의 완료, 런타임 검증 미구현 |
| 스킬 (skills/) | 🟡 중간 | run 스킬 스모크 테스트 완료, 나머지 3개 미검증 |

## 6. ⚠ 함정 및 주의사항

**담당 Performer**: devils-advocate

### 6.1 과잉설계 경고

- **7개 에이전트 × 5단계 파이프라인**은 소규모 변경에는 과잉이다. bugfix 프리셋이 4에이전트+R1만으로 경량화했지만, 실제 사용 시 "한 줄 수정에 7명이 토론" 상황이 발생할 수 있다
- **Reuse-First 4개 디바이스**는 개념적으로 우아하지만, 실제 소규모 프로젝트에서 "재사용 인벤토리"가 비어있으면 의미 없는 의식(ceremony)이 된다

### 6.2 아직 증명되지 않은 가정

| 가정 | 리스크 |
|---|---|
| Ollama 로컬 모델이 에이전트 역할을 충분히 수행 | llama3.1:8b가 QA 역할을 잘 해내는지 실전 미검증 |
| 합의 70% 임계치가 적절 | 실전 데이터 없이 설정된 임의 수치 |
| Gemini 무료 티어 rate limit이 충분 | 다수 라운드 시 rate limit 도달 가능성 |
| scribe가 6개 섹션을 일관성 있게 통합 | 실전에서 용어 불일치가 얼마나 심한지 미확인 |

### 6.3 "고치지 마" 영역

- **CONTRACT.md**: 33KB 오라클 문서. 수정 시 전체 에이전트 정의와의 정합성 검증 필요. 부분 수정 금지 — 수정하려면 전체를 이해한 후 해야 함
- **INTERVIEW.md**: Gate1 설계 결정 기록. 이미 확정된 결정을 뒤집으려면 새로운 Gate를 열어야 함
- **에이전트 frontmatter의 model/tools 필드**: 스키마 계약의 일부이므로 임의 변경 시 파이프라인 깨짐

### 6.4 반직관적 포인트

- **scribe는 토론에 참여하지 않는다**: 문서 기록자일 뿐. Peer Signature도 없고 Plan 수정 불가
- **devils-advocate만 Reuse-First auto-disagree 면제**: 다른 모든 Performer는 "new" 결정 시 자동 반대 투표를 받지만, devils-advocate는 재사용 강제에 정당하게 반론 가능
- **Phase 2는 외부 Performer가 실행하지 않는다**: Claude Code(Conductor)만 파일 수정 권한 보유. 이것은 보안 설계 결정
- **Deep Scan 6항목 미완시 Phase 1 진입 불가**: 선택 4항목은 건너뛸 수 있지만, 필수 6항목은 차단 조건

### 6.5 교훈 (Lessons Learned)

- Gate1에서 ChatGPT 트랜스포트를 검토했으나 ToS 안정성 문제로 제외 → 향후 재검토 불필요
- 스킬 디렉토리 이름에 `ensembra-` 접두사를 붙였다가 Gate2에서 제거 (커밋 `270e829`) → 네임스페이스는 `ensembra:` 콜론 접두사로 충분
