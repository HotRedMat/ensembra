## 0. 요약

`agents/` 디렉토리는 Ensembra 멀티에이전트 토론 파이프라인의 **역할 선언문 8개**를 담고 있다. 1 Conductor(orchestrator) + 6 Performer(planner, architect, developer, security, qa, devils-advocate) + 1 Recorder(scribe) 구성이며, 각 파일은 YAML frontmatter + Markdown body로 역할·책임·출력 계약·금지사항을 정의한다. 코드가 아닌 **선언적 설정**이므로 빌드 없이 Claude Agent tool이 직접 참조한다. Gate2까지 완료, 런타임 실행 로직은 Gate2에서 구현되었다.

---

## 1. 프로젝트 목적·목표

- **목적**: 6명의 전문 에이전트가 토론·합의를 거쳐 코드 변경 Plan을 만들고, Claude Code 본체가 실행하는 파이프라인의 **역할 정의**
- **목표**:
  - 각 에이전트의 의도(mission)와 경계(scope) 명확화
  - Conductor-Performer 간 입출력 계약 (`schemas/agent-output.json`) 준수
  - Reuse-First 교차 원칙을 모든 Performer에 내재화
- **열린 요구사항**:
  - [ ] frontmatter의 model/tools 가 실행 시점 바인딩과 일치하는지 검증
  - [ ] CONTRACT.md 와의 drift 정기 점검 (Gate2 self-audit에서 4건 수정 이력)

## 2. 아키텍처

### Conductor-Performer 패턴

| Role | 유형 | 기본 Transport | 폴백 | Phase 참여 |
|------|------|---------------|------|-----------|
| orchestrator | Conductor | Claude opus | — | 0~4 전체 조율 |
| planner | Performer | Claude opus | — | 1, 3 |
| architect | Performer | Gemini 2.0-flash | Claude sonnet | 1, 3 |
| developer | Performer | Claude sonnet | — | 1, 3 |
| security | Performer | Ollama qwen2.5:14b | Claude sonnet | 1, 3 |
| qa | Performer | Ollama llama3.1:8b | Claude sonnet | 1, 3 |
| devils-advocate | Performer | Claude haiku | — | 1, 3 |
| scribe | Recorder | Claude sonnet | — | 4 전용 |

### 파이프라인 흐름

```
Phase 0: Deep Scan (강제 6항목 + 선택 4항목)
Phase 1: Deliberate — R1 → [R2 Peer Signature] → Synthesis
Phase 2: Execute — Claude Code 본체만 (외부 Performer 쓰기 금지)
Phase 3: Audit — 감사자 검증, Rework 최대 2회
Phase 4: Document — scribe 전용 (Task Report 강제)
```

### 핵심 설계 결정

- **Transport 다원화**: 역할별 인지 패턴에 맞는 모델 배정 (구조분석→Gemini, 체크리스트→Ollama 로컬)
- **Reuse-First 4단 장치**: 스키마 필드 강제 → 자동 disagree → Synthesis 섹션 강제 → devils-advocate 예외
- **Scribe 격리**: Phase 4 전용, 토론·투표 불참 → 기록자 편향 방지
- **Phase 2 권한 일원화**: Claude Code 본체만 파일 수정 → 실행 추적성 확보

### 모듈 의존성

- 전 에이전트 → `CONTRACT.md` (정본), `schemas/agent-output.json` (출력 스키마)
- orchestrator → 모든 Performer (입력 분배·출력 수집)
- scribe ← 6 Performer 출력 (Phase 4 입력)

## 6. ⚠ 주의할 함정

> 신규 기여자가 agents/ 작업 시 반드시 숙지해야 할 함정 목록.

1. **frontmatter model ≠ 실제 Transport**
   - `architect.md` frontmatter `model: sonnet`은 **폴백**용. 실제 기본은 Gemini 2.0-flash
   - `orchestrator.md` 본문의 Performer 풀 설명과 각 에이전트 frontmatter가 다를 수 있음
   - **규칙**: 실제 Transport는 `CONTRACT.md` §8 + `config.json`이 정본

2. **scribe ≠ 일반 Performer**
   - agents/ 에 동급으로 나열되어 있지만, scribe는 토론 불참·Peer Signature 대상 아님
   - Phase 4에서만 활성화되는 **기록자** — Performer 목록에 포함시키면 합의율 계산이 틀어짐

3. **devils-advocate의 Reuse-First 예외**
   - 유일하게 자동 disagree 규칙에서 면제. 재사용 강제에 반대하는 것도 합법적 의견
   - 다른 Performer에 같은 예외를 부여하면 Reuse-First 체계가 무력화됨

4. **CONTRACT.md ↔ agents/ drift**
   - 모든 에이전트가 CONTRACT.md를 정본이라고 선언하지만, 실제로 drift 발생 이력 있음
   - Gate2 self-audit (커밋 `40c0fce`)에서 4건 수정됨. 변경 시 양쪽 동기화 필수

5. **Phase 2 쓰기 권한은 Claude Code 본체 전용**
   - 외부 Performer(Gemini, Ollama)에게 파일 쓰기를 위임하면 추적성이 깨짐
   - 이 원칙은 보안·감사 양쪽에서 강제됨

6. **tools 필드 순서 불일치**
   - orchestrator: `Read, Glob, Grep` / 나머지: `Read, Grep, Glob`
   - 기능에 영향 없으나, 일괄 정리 시 diff 혼란 유발 가능. 의도적 차이 아님

7. **해당 없음 영역** (프로젝트 초기라 누적 부족)
   - 과거 시행착오 DB 미축적 — Task Report 누적 후 이 섹션이 풍부해짐
